# IronCircuit Blueprint
## 1. Project Overview
IronCircuit is a C++ powered circuit breaker for LLM agents. It moves the responsibility of token monitoring out of the slower Python layer and into a compiled, high-performance C++ core. It is designed to be "fail-closed," meaning if the system cannot verify your token budget, it defaults to blocking the agent, preventing unauthorized cost overruns.

## 2. Technical Architecture
The system operates as a hybrid bridge:
* The Engine (C++): A memory-resident monitor that performs integer-based arithmetic for token tracking. This ensures microsecond latency, regardless of how complex your Python agent becomes.
* The Bridge (pybind11): A lightweight interface that binds the C++ core to Python, allowing your agent to "protect" any function via a simple decorator.
* Persistence Layer: A JSON-backed, thread-safe storage system that records usage to disk. It uses strict file locking to ensure that even under high concurrency, your token count remains accurate and corruption-proof.

## 3. The Problems It Solves
* Runaway Costs: Agent loops can unintentionally call expensive APIs thousands of times. IronCircuit caps this at the hardware-logic level.
* State Corruption: Most Python-based monitors fail if a file is partially written during a crash. IronCircuit includes robust error handling to reset to a known-safe state if a state.json file is malformed.
* Concurrency Gaps: Standard file I/O isn't thread-safe. IronCircuit uses threading.Lock mechanisms to ensure that parallelized agent threads do not overwrite each other’s token counts.

## 4. Verification & Testing Methodology
To ensure this project is "flawless," we employed a three-tier testing strategy:
1. Unit Validation: Verified the C++ logic handles floating-point token costs correctly and implements a strict "fail-closed" gatekeeper.
2. Concurrency Stress Test: We simulated 10 concurrent threads hammering the monitor simultaneously to verify that the threading.Lock prevented data races. The system successfully maintained an accurate, cumulative total of 150.0 tokens under load.
3. Resilience Testing: We verified that if state.json is missing or corrupted, the system does not crash the host application. Instead, it logs a warning and safely initializes a new baseline, ensuring high availability.
